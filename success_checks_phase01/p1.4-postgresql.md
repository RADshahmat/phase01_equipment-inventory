id SERIAL PRIMARY KEY,

3. orders
id SERIAL PRIMARY KEY,
user_id INT REFERENCES users(id),
product_id INT REFERENCES products(id),
created_at TIMESTAMP DEFAULT NOW()
## Sample Data Flow
Users placed orders
Each order is linked to:
one user
one product

This creates a many-to-many relationship via orders table

🔗 Three-Way JOIN Query
SELECT 
  u.name,
  p.name,
  o.created_at
FROM orders o
JOIN users u ON o.user_id = u.id
JOIN products p ON o.product_id = p.id;
⚡ Performance Optimization (Indexing)
Index Added
CREATE INDEX idx_orders_user_id ON orders(user_id);
🧪 Query Analysis Using EXPLAIN
After Index Execution Plan
Nested Loop  (cost=1.27..44.68 rows=5 width=72)
   ->  Hash Join  (cost=1.11..19.83 rows=5 width=44)
         Hash Cond: (u.id = o.user_id)
         ->  Seq Scan on users u  (cost=0.00..16.30 rows=630 width=36)
         ->  Hash  (cost=1.05..1.05 rows=5 width=16)
               ->  Seq Scan on orders o  (cost=0.00..1.05 rows=5 width=16)
   ->  Index Scan using products_pkey on products p
         Index Cond: (id = o.product_id)
         
## Interpretation
1️⃣ users table
Seq Scan on users
Small table or not indexed filtering needed
PostgreSQL scans all rows

2️⃣ orders table
Seq Scan on orders → Hash Join
Orders scanned once
Hash table created for join optimization

3️⃣ products table
Index Scan using products_pkey
Primary key index used
Fast direct lookup instead of full scan

⚙️ Key Observations
PostgreSQL optimizer uses a Hybrid strategy
Sequential Scan for small datasets
Hash Join for efficient joining
Index Scan for primary key lookup
Index improves targeted lookups significantly
Join performance depends on:
FK columns
indexed fields
